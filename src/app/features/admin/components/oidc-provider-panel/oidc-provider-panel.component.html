What is this?

qb-engineer can act as an OpenID Connect (OIDC) Identity Provider for other applications. Employees sign in to qb-engineer once and can reach authorized third-party apps without separate passwords. This panel is where you onboard those apps, approve them, and monitor their activity.

info
When to use this: you have an external app (internal tool, vendor portal, reporting dashboard, etc.) that needs to know who the signed-in qb-engineer user is. Instead of managing a second set of passwords, let that app delegate authentication here. You stay in control of who can access it via role gates and consent screens.

The three-step onboarding flow

  1. Mint a registration ticket — click Mint ticket in the top-right. Enter the expected client name (human identifier), the redirect URI prefix the client will use (e.g. https://reports.mycorp.com/auth/), the scopes you want to allow, and an expiration (default 48 hours). You'll get a one-time code that looks like qbetkt_<prefix>_<random> — copy it once; it won't display again.
  2. Share the ticket out-of-band — send the code to the client developer via a secure channel (encrypted chat, password manager share, signed email). The ticket is single-use; redeeming it once invalidates it.
  3. Client self-registers, then you approve — the client's server POSTs to /connect/register (RFC 7591) with the ticket plus its metadata (display name, logo, redirect URIs inside the allowed prefix, optional signed software statement). qb-engineer creates a Pending client and shows it on the Clients tab. Click the client to review what was submitted, then choose first-party vs. third-party, auto-grant vs. consent-required, which roles may authenticate, and which custom scopes they may request. Approving flips the status to Active and the client can begin using /connect/authorize immediately.

First-party vs. third-party (what you're choosing on approval)

First-party

Apps your org owns — employee tools, internal dashboards, custom software. Default to auto-grant consent because users shouldn't be asked to approve internal apps every login.

Typical icon: your company's own logo. Recommend role-gating to employees only.

Third-party

Apps external vendors own that want to sign users in via qb-engineer. Default to require consent — the user sees a screen listing exactly what data the app will receive.

Users can revoke third-party apps from Account › Connected Apps at any time.

The three-tier rotation model

There are three distinct secrets in this system. Each rotates at a different moment, and you never need to touch two at once.

SecretWho holds itWhen it rotates
Registration ticket
qbetkt_…		 Admin → shared once with client developer Consumed on first /connect/register call, then invalid forever
Client secret
qbecs_…		 Client server (stored in client's config) Rotated from client detail dialog → Rotate secret. Old secret invalid on rotation.
Registration access token
qberat_…		 Client server (used for self-service metadata updates) Automatically rotated each time the client PATCHes its own /connect/register/{client_id}

Scopes — what claims the client receives

Scopes are named permissions the client requests at /connect/authorize. Built-in scopes follow the OpenID Connect spec:

  • openid — required; subject identifier (user ID)
  • profile — name, picture, locale
  • email — email + email_verified
  • roles — qb-engineer roles array
  • offline_access — refresh token (client can renew tokens without user present)

Custom scopes live on the Scopes tab. Each custom scope maps a JSON object into token claims — e.g. {"qb_company_id":"CompanyId","qb_permissions":"Permissions"} adds those fields to the id_token and /connect/userinfo response. Keys are claim names; values are either literal strings or {PropertyName} refs into the user profile.

Endpoints (for the client developer)

  • /.well-known/openid-configuration — discovery document; point the client library here and it auto-configures.
  • /.well-known/jwks — signing keys (JWKS). Clients fetch this to verify JWT signatures.
  • /connect/register — RFC 7591 dynamic client registration (POST with ticket).
  • /connect/register/{client_id} — client reads/updates its own metadata using the registration access token.
  • /connect/authorize — interactive login + consent screen (GET; redirects back to client with code).
  • /connect/token — code exchange + refresh token grant (POST).
  • /connect/userinfo — claims for the signed-in user (GET with bearer token).
  • /connect/logout — end-session endpoint; clears qb-engineer session and redirects back to client.

Security controls in place

  • Authorization Code + PKCE only. Implicit flow is not supported; it's deprecated for good reason.
  • Three-tier secret rotation. See above. Compromised client? Rotate the client secret; no user or ticket impact.
  • Optional signed software statements. Add a publisher's JWKS URL to the ticket and the client must present a signed statement — prevents ticket theft from being useful without the publisher's private key.
  • Role gates per client. Only users whose role is on the client's allow-list can sign in. A denied attempt is logged in the audit trail with event RoleGateDenied.
  • Per-client consent screens. Third-party apps get an explicit consent UI; users can revoke from Account › Connected Apps. The grant is persisted per user per client.
  • Redirect URI strict-prefix matching. The client must redirect to a URL matching the prefix you set on the ticket; any mismatch is rejected and logged as RedirectUriMismatch.
  • Full audit log covers every admin action (ticket mint, client approve/suspend/revoke, secret rotate, scope changes) and every runtime event (tokens issued, consent granted/denied, security violations).

Common mistakes to avoid

  • Sharing the ticket over email. Plain email is not secure. Use an encrypted chat or a password manager share. If a ticket leaks, revoke it from the Tickets tab.
  • Making every client first-party. First-party auto-grants consent. If the app is not yours, users deserve a chance to see what it's taking. Default to third-party unless you control the codebase.
  • Leaving redirect prefix too broad. https:// matches any HTTPS URL. Narrow it to the app's actual host + path: https://reports.mycorp.com/auth/.
  • Rotating the client secret while the client is live. The old secret invalidates immediately on rotation; coordinate with the client developer. Existing user sessions continue (they don't use the secret), but the client can't exchange new codes until it has the new secret.
  • Revoking instead of suspending. Revoke is terminal — you cannot un-revoke. For a temporary issue (compromise suspicion, payment lapse, maintenance window), suspend. Suspend → Approve restores the client with the same client_id and secret.

Troubleshooting

Client says "ticket invalid or expired"
Tickets are single-use and time-boxed. Check the Tickets tab — if the status is Expired, mint a new one. If Redeemed, the client already registered; look on the Clients tab.
User sees "access denied" at /connect/authorize
Role gate. Check the client detail dialog → Allowed roles. If the user is not in one of those roles, either add their role to the client or grant them the role. The audit trail will have a RoleGateDenied event.
Client says "redirect_uri mismatch"
The client's actual redirect URI must start with the prefix recorded when they registered. Check RedirectUriMismatch events in the audit trail — the attempted URI is in the payload.
Consent screen appears every login for a first-party app
Open the client detail, flip Require consent off. First-party apps typically auto-grant.
Custom claims aren't showing up in the token
Confirm three things: (1) the scope is marked Active, (2) the client's Allowed custom scopes includes it, (3) the user's profile actually has the property referenced in the mapping. Empty properties become null claims.
